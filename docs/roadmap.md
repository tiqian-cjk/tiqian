# 提椠 Roadmap

这张表记录提椠已经完成的实施切片、当前工作和后续候选项。状态字段只在工作范围确实变化时更新：

- `done` — 已合入 main 且有 fixture + 测试覆盖。
- `wip` — 有 commit 但还没满足验收清单。
- `todo` — 还没开始。

当前 pipeline 与模块边界见 [architecture.md](architecture.md)，排版取舍见
[ADR 索引](adr/README.md)。中文排版研究背景见 [research/kongque-notes.md](research/kongque-notes.md)，
禁则档与悬挂随行长变化的实测见
[research/kinsoku-hanging-line-width-experiment.md](research/kinsoku-hanging-line-width-experiment.md)。

## 当前位置

当前主线是继续在真实站点与应用中使用 Compose / Web 前端，由实际问题决定后续切片。
Slice 35 已完成；目前没有为了占位而开启新的 `wip`。

已经命名但尚未开工的候选：

- `WidthIndependentAnnotationCache`：让不依赖行宽的 annotation 结果在 resize 重排时复用。
- `OpaqueInlineObjectGeometryInvalidation`：图片或静态 inline object 加载后几何变化时重新测量。

最近完成的是 **Slice 35：Web 真实站点宿主接入**（2026-07-11）。`@tiqian/web` 以
ESM 包和 light-DOM `<tiqian-prose>` 接入真实博客；SSR、无 JavaScript、Pagefind、宿主 CSS、
源忠实复制与客户端导航生命周期均有自动化测试和真实浏览器验证。完整取舍见
[ADR 0039](adr/0039-web-rendering-path.md)。

Slice 15 的依据（CLREQ 原文）：

> 「与西文排版不同，中文排版特别是书籍正文排版极少使用左齐右不齐，原则上
> 应该进行两端对齐。西文排版两端对齐（justification）时，主要是调整单词
> 之间的间隙（词距），而中文排版在两端对齐时，能调整的地方更多，具体如下
> 所述。」

——双齐不是可选项而是基线，对齐的自由度只在末行（齐左/居中/齐右）。

完成行保留当时的目标、API 名称和验收方式，是实施记录，不是当前 API 参考；后续重命名或
修订以相关 ADR 和当前代码为准。

## 当前范围之外

- 简体与繁体中文直排。
- 日文 JLREQ 与韩文 KLREQ profile。
- 分页、多栏、脚注等页面布局。
- 编辑器、IME 与完整 selection 几何。
- 完整 CSS Text 兼容。

这些能力需要新的 writing mode、页面模型或前端契约，不应作为现有横排实现上的局部补丁加入。

## 怎么用这张表

- 普通 bug 修复、测试和文档修正不需要新建 Slice。
- 需要跨提交持续跟踪的新工作，先定义目标、验收语料和命令，再新增 `todo` 行。
- 开工后改为 `wip`；实现、测试与人工验收完成后才改为 `done`。
- 有意改变公共 API、模块边界或既有排版取舍时，同步更新或新增 ADR。
- 一个 slice 跨多个 PR 是正常的——但只有验收 fixture + 测试 + dump 三件都满足才能改 `done`。
